# 系统架构升级方案

## 升级原则

1. 保持现有页面、接口字段和业务流程兼容，架构调整不与业务功能改动混在同一批发布。
2. 所有数据库结构变化必须有唯一版本号、执行记录和可验证结果，禁止继续堆叠无记录的启动脚本。
3. 正式环境业务数据只进入 RDS MySQL，文件只进入 OSS；ECS 仅运行无状态应用和保留兼容性缓存。
4. 每批改造必须通过前后端构建、关键规则测试、本地接口验证和正式环境健康检查。
5. 优先拆分高频变更区域，避免一次性重写造成业务回归。

## 当前架构与主要风险

当前系统采用 React + Vite 前端、Express + MySQL 后端、RDS MySQL、OSS 和 Docker 部署。技术选型适合当前业务规模，但代码经历多轮快速迭代后出现以下结构性风险：

- `OrdersListPage.tsx`、`CustomersListPage.tsx`、`AppointmentsCalendarPage.tsx` 同时承担查询、状态、业务规则、表格和弹窗，修改一个字段容易影响多个流程。
- 后端路由直接包含查询、转换、权限和业务事务，缺少 service/repository 边界。
- 历史数据库升级逻辑在每次启动时重复执行，缺少迁移版本记录。
- 客户池、订单快照、排期服务次数等关键规则分散在前后端，容易产生口径不一致。
- 自动化测试和持续集成覆盖不足，问题主要依靠人工回归发现。
- 大量 `any` 和重复 JSON/日期解析降低类型保护能力。

## 目标分层

```text
前端页面
  -> 页面模块（列表、编辑器、详情）
  -> 业务 hooks 与领域规则
  -> 类型化 API 客户端

后端路由（鉴权、参数校验、响应）
  -> Service（客户转单、服务进度、跟进记录）
  -> Repository（事务与 SQL）
  -> RDS MySQL / OSS
```

## 分阶段实施

### 第一阶段：稳定性地基

- 引入 `schema_migrations`，数据库变更只执行一次并留下记录。
- 增加数据库就绪检查和连接优雅关闭。
- 正式环境关闭 ECS 本地文件回落，OSS 异常时明确失败，避免文件散落。
- 建立统一序列化工具、关键基础规则测试和 GitHub 质量门禁。
- 为客户池列表和排期查询增加组合索引，降低数据增长后的 RDS 扫描开销。

### 第二阶段：后端业务分层

- 先拆订单和客户：route 只做协议处理，service 管业务，repository 管 SQL 和事务。
- 增加轻量字典/选择器接口，替代页面一次拉取 1000 到 10000 条完整记录的做法。
- 将“客户池转订单池”改为单一事务，并为同一客户的有效订单增加数据库约束或明确的业务幂等键。
- 将跟进状态、服务次数、人工校正优先级集中为领域服务，前端只展示服务端结果。
- 为写接口增加 Joi 请求校验和统一错误码。

### 第三阶段：前端模块化

- 将订单编辑器拆为客户信息、订单维护、服务人员、跟进记录四个独立模块。
- 将客户列表的筛选、表格、编辑弹窗、导入导出拆分，并抽取共用客户档案组件。
- 收紧 API 类型，逐步清除核心流程中的 `any`。
- 对大页面采用按路由加载，降低首屏包体积。

### 第四阶段：回归与可观测性

- 覆盖登录权限、客户转单、预约完成、服务次数、上传 OSS、删除确认等端到端测试。
- 增加结构化日志、请求标识、慢查询记录和错误告警。
- 建立数据库备份恢复演练、部署回滚和迁移前快照流程。

## 每次迭代的完成标准

- 前后端构建通过，基础测试通过。
- 新增或修改的业务规则有自动化测试。
- 数据库变更有独立迁移编号，不在路由或启动函数中临时修改表结构。
- 管理员、客服顾问、技师权限至少各验证一次。
- 本地验证通过后再合并 GitHub，并在备份完成后部署正式环境。
- 部署后检查 `/health`、`/ready`、登录、RDS 查询和 OSS 上传。

## 已完成进度

### 第一阶段

- 已建立有版本记录的数据库迁移、就绪检查和数据库连接优雅关闭。
- 已关闭正式环境 OSS 异常时的 ECS 本地文件回落。
- 已加入基础序列化规则测试、组合索引和 GitHub 质量检查。

### 第二阶段（订单核心写入）

- 订单新建、编辑和删除已从路由拆入独立业务服务。
- “客户池转订单池”已改为单一数据库事务：订单保存失败时客户资料不会被移除。
- 同一客户重复建单和重复订单号统一返回冲突提示，不再产生重复数据。
- 客户编号生成同时检查客户池和订单快照，客户转入订单池后原编号不会被新客户重复使用。
- 审计日志只记录真正成功的写操作，失败请求不会被记成成功操作。
- 订单付款状态兼容规则已加入自动化测试。

大列表的全量查询优化将作为独立批次推进。当前多个报表依赖完整客户和订单字段，需先补充服务端筛选、汇总和轻量选项接口，再逐页替换，避免一次性改动影响现有使用。

### 第三阶段（客户列表查询）

- 客户列表已从每次加载 10000 条完整记录改为服务端筛选和分页，默认每页 20 条，并兼容 10、30、50 条切换。
- 关键词、获客时间、区域、来源、跟进状态、标签和归属客服统一由后端组合查询，列表总数与分页结果使用同一筛选口径。
- 客户关键词输入增加短时防抖，连续输入不会对 RDS 逐字触发查询。
- 管理员和超级管理员的客户导出改为独立受控接口，导出当前筛选条件下的全部记录，不受当前页限制；其他角色由后端拒绝访问。
- 增加轻量客服选项接口，筛选下拉不再依赖当前页客户数据，也无需为选项加载全量客户资料。
- 客户池边界集中在查询服务中：默认只返回 D1-D3 且尚未转单的客户，旧接口参数继续兼容。
- 客户查询组合条件已加入自动化测试，并通过本地 MySQL 的分页、筛选、导出一致性和角色权限回归。

订单列表、报表和部分选择器仍存在 1000 到 10000 条兼容性查询，后续按页面逐批迁移；在依赖页面完成替换前，不对通用接口强制设置全局上限。

### 第四阶段（前端加载与回归基线）

- 登录页与 13 个业务页面改为按需加载，首个主脚本由约 1.10 MB 降至约 291 KB，构建不再出现单文件超过 500 KB 的警告。
- 页面切换统一增加加载状态，并保留页面级错误边界；单个业务页面加载失败不会拖垮整个应用框架。
- 新增只读全局接口回归脚本，覆盖健康检查、数据库就绪、未登录保护、客户、订单、预约、技师、服务记录、合同、财务、首页看板、用户、操作日志和客服角色权限边界。
- 本地页面回归覆盖首页、客户列表、订单列表、数据报表、技师档案、排期管理、预约列表、工资结算、收支管理和系统设置，所有页面均完成按需加载且浏览器控制台无错误。
- API 客户端、端点和 React Query 钩子已统一查询参数、错误响应和看板数据契约，清除该公共层 47 个宽泛类型错误。
- 当前全量 ESLint 基线为 35 个文件、244 个历史错误和 22 个警告，主要集中在旧大页面、演示兼容层和后端旧路由；后续按模块清理，不以关闭规则或批量重写掩盖债务。

只读接口回归通过 `REGRESSION_BASE_URL`、`REGRESSION_ADMIN_TOKEN` 和可选的 `REGRESSION_SERVICE_TOKEN` 注入测试环境，运行 `npm run regression:api`。脚本不内置账号、密码或密钥，也不写入数据库。
